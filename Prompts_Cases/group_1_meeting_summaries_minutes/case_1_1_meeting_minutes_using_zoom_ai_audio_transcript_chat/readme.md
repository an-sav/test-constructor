# Анализ промпта

## Ключевые компоненты промпта

* **Персона:** "Аналитик Sanofi" со специализацией на итогах деловых встреч.
* **Контекст:** Работа в корпоративной среде Sanofi с использованием данных из Zoom AI (транскрипты, видео, аудио, чаты).
* **Основная цель:** Генерация структурированных отчетов (summary) по итогам встреч.
* **Входные данные:** Записи встреч Zoom, название и дата встречи (предоставляются пользователем).
* **Ключевые технологии:** Упоминание специфических инструментов (Zoom AI, `searchUsers`, `orgHierarchy`, `knowledgeBaseSearch`).
* **Пошаговая логика:** Четкая последовательность действий: подготовительный этап, анализ по 8 обязательным компонентам.
* **Строгий формат вывода:** Детально описанная структура отчета с разделами и Markdown-таблицами (задачи, RACI).
* **Ограничения:** Запрет на использование неподтвержденных данных, интерпретаций и раскрытие чувствительной информации.
* **Тональность:** Профессиональный, нейтральный, с использованием бизнес-лексики Sanofi.
* **Интерактивность:** Готовность задавать уточняющие вопросы и сообщать об отсутствующих данных.

## One-pager / Краткое описание

* **Основное назначение (1 предложение):** Промпт превращает LLM в корпоративного аналитика, который автоматически создает подробные отчеты по итогам деловых встреч на основе данных из Zoom.
* **Цель (2-3 предложения):** Цель — стандартизировать и автоматизировать процесс подведения итогов встреч в компании Sanofi. Промпт предназначен для сотрудников, которым необходимо быстро получать структурированную информацию о принятых решениях, назначенных задачах и последующих шагах, экономя время на ручной обработке записей.
* **Ключевые инструкции:** LLM должен принять роль аналитика Sanofi, верифицировать участников через внутренние инструменты, проанализировать все доступные материалы встречи (транскрипт, чат, аудио), извлечь ключевые пункты (решения, задачи, риски) и оформить их в виде строго структурированного отчета.
* **Ожидаемый результат:** Готовый к использованию документ, содержащий сводку встречи, иерархию тем, список решений, таблицу задач, RACI-матрицу, перечень неясностей, план follow-up действий и оценку качества коммуникации.

## Архитектура и логика промпта

Промпт построен по принципу "сверху вниз" — от общей роли и цели к конкретным инструкциям и ограничениям. Он использует блочную структуру с заголовками (`# РОЛЬ`, `# ЦЕЛЬ` и т.д.), что упрощает его чтение и интерпретацию моделью. Логика выполнения — последовательная: сначала подготовка и верификация данных, затем основной анализ по заданным пунктам, и в конце — форматирование вывода в соответствии с шаблоном.

**Диаграмма логического потока:**

```mermaid
graph TD
    A[Начало: Пользователь предоставляет данные о встрече] --> B{Этап 1: Подготовительные действия};
    B --> B1[Проверка участников через `searchUsers` и `orgHierarchy`];
    B --> B2[Анализ всех источников: транскрипты, аудио, чат];
    
    subgraph "Этап 2: Основной анализ"
        direction RT
        C["a) Выделение тем"]
        D["b) Фиксация решений"]
        E["c) Формулировка задач"]
        F["d) RACI-анализ"]
        G["e) Определение неясностей"]
        H["f) Определение Follow-up"]
        I["g) Выявление рисков"]
        J["h) Шаблон email"]
    end
    
    B1 & B2 --> C & D & E & F & G & H & I & J;
    
    J --> K{Этап 3: Форматирование вывода};
    K --> L[Сборка отчета по разделам a-h с таблицами];
    L --> M[Конец: Выдача структурированного отчета];

```

## Карта инструкций (Mapping Instructions → Behavior)

| Инструкция из промпта                                                   | Ожидаемое поведение LLM                                                                                                             | Комментарий                                                                        |
| :---------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------- |
| `Вы аналитик Sanofi, специализирующийся на анализе...`                  | LLM принимает заданную профессиональную роль, использует соответствующий словарный запас и подход к анализу.                        | Это ключевая инструкция, задающая персону и контекст.                              |
| `Использовать функцию searchUsers для верификации...`                   | LLM понимает, что ему необходимо обращаться к внешнему инструменту (или симулировать его работу) для проверки данных об участниках. | Промпт предполагает наличие у LLM специальных возможностей.                        |
| `Провести RACI‑анализ участников`                                       | LLM применяет методологию RACI для распределения ролей и ответственности на основе содержания дискуссий.                            | Требует от модели не просто извлечения фактов, а их анализа и классификации.       |
| `                                                                       | Задача \| Ответственный \| ... \|`                                                                                                  | LLM форматирует извлеченные задачи в виде Markdown-таблицы с указанными колонками. | Четкая инструкция по форматированию вывода. |
| `Не допускать интерпретаций, не основанных на содержании встречи`       | Модель придерживается исключительно фактов из предоставленных материалов, избегая домыслов и "галлюцинаций".                        | Это важное ограничение для повышения точности и объективности.                     |
| `При отсутствии данных — укажите это и предложите запросить информацию` | LLM проявляет проактивность: не придумывает недостающую информацию, а сообщает о ее отсутствии.                                     | Повышает надежность и интерактивность модели.                                      |

## Разбор персоны и тональности

* **Роль:** Внутренний аналитик компании Sanofi. Это подразумевает знание корпоративной культуры, стандартов и бизнес-процессов.
* **Знания и экспертиза:** Модель должна обладать экспертизой в следующих областях:
  * Анализ деловых коммуникаций.
  * Методологии управления проектами (RACI, постановка задач).
  * Использование инструментов на базе ИИ (Zoom AI).
  * Корпоративная этика (конфиденциальность, точность).
* **Тон и стиль:** Строго профессиональный, нейтральный, объективный и структурированный. Язык должен быть формальным, с использованием принятой в Sanofi бизнес-терминологии. Эмоциональные оценки или субъективные мнения исключены.
* **Ограничения персоны:** Персона не имеет права:
  * Делать выводы, не подтвержденные фактами из записи встречи.
  * Раскрывать конфиденциальную информацию.
  * Игнорировать прописанную структуру отчета.
  * Давать собственные оценки или мнения.

## Анализ рисков и неоднозначности

* **Неясные инструкции:** Промпт ссылается на вымышленные или недоступные для LLM инструменты (`searchUsers`, `orgHierarchy`, `knowledgeBaseSearch`). Модель не может их выполнить и будет либо "галлюцинировать" их вывод, либо сообщит об ошибке. Неясно, как именно модель должна получить доступ к "видео, транскриптам, аудио и чатам" — как единому пакету данных или по отдельности.
* **Потенциальные "галлюцинации":** Сложность анализа (например, "оценка качества коммуникации" или "уровень согласованности решений") может спровоцировать модель на генерацию умозрительных выводов. Без четких метрик эти оценки могут быть субъективными и не соответствовать действительности, даже если модель будет пытаться основываться на транскрипте.
* **Конфликтующие правила:** Прямых конфликтующих правил нет, однако существует внутреннее противоречие между требованием `отражать только факты` и задачей `оценить качество коммуникации`, которая по своей природе является интерпретационной.
* **Рекомендации по снижению рисков:**
    1. **Уточнить работу с инструментами:** Переформулировать инструкции. Вместо "Использовать `searchUsers`" написать: "В начале работы запроси у пользователя список участников с их должностями и департаментами в формате таблицы".
    2. **Декомпозировать сложные задачи:** Вместо общей "оценки качества коммуникации" дать более конкретные инструкции: "Подсчитай количество участников, которые высказывались. Зафиксируй моменты, где обсуждение уходило от темы повестки".
    3. **Определить формат входных данных:** Четко указать, в каком виде пользователь должен предоставить материалы встречи (например, "Вставьте полный текст транскрипта ниже").

## Рекомендации по улучшению промпта

1. **Добавить "Chain-of-Thought" (Цепочка размышлений):** Внедрить предварительный шаг, где модель сначала формирует план анализа. Добавить инструкцию: "Перед тем как генерировать отчет, в тегах `<planning>` кратко опиши свой план: какие ключевые темы ты выделил, какие решения и задачи зафиксировал. Только после этого приступай к финальному отчету". Это повысит структурность и точность вывода.
2. **Предоставить Few-Shot пример:** Добавить в секцию "ФОРМАТ ВЫХОДНЫХ ДАННЫХ" небольшой, но реалистичный пример заполненной таблицы или раздела. Например, показать, как выглядит готовый RACI-анализ для одной из тем. Это даст модели четкий образец для подражания.
3. **Ввести механизм обработки ошибок:** Явно прописать, что делать, если транскрипт неполный или содержит ошибки. Например: "Если в транскрипте есть неразборчивые фрагменты, обозначь их как `[неразборчиво]` и не пытайся восстановить их смысл. Если из-за этого невозможно выполнить какой-либо пункт анализа, укажи на это в соответствующем разделе".
4. **Усилить ограничения по интерпретации:** Для таких пунктов, как "оценка качества коммуникации", явно потребовать аргументацию. Например: "Твоя оценка вовлеченности участников должна быть подкреплена цитатами или фактами из транскрипта (например, 'Иванов А. предложил 3 идеи, Петров Б. задал 5 уточняющих вопросов')".

## Лог итераций (Внутренний)

* **Итерация 1:** Сгенерирован черновик отчета. Основные компоненты промпта (персона, цель, формат, ограничения) успешно определены и структурированы. Выявлен основной риск, связанный с недоступностью внешних инструментов.
* **Итерация 2:** Добавлена диаграмма Mermaid для визуализации логики. Углублен анализ рисков, добавлены пункты о сложности анализа и конфликте между фактами и оценкой. Сформулированы финальные, более конкретные и actionable рекомендации по улучшению промпта. Откорректирована формулировка в "Карте инструкций" для большей ясности.
